# Supervisor demo guide — disagreement-aware X2DFD layer

A one-page script for the meeting. Five minutes if everything works, ten if
questions land on the rules.

---

## 1. The problem this POC demonstrates

A single X2DFD verdict + rationale comes from **one model run with one expert
configuration**. There is no built-in way to tell whether that verdict would
survive removing an expert, swapping experts, or whether the confidence is
actually high enough to act on.

The POC adds a **disagreement-aware comparative explanation layer**: it runs
the same image under four expert settings (`none` / `blending` / `diffusion`
/ `blending+diffusion`), compares the four resulting JSONs, and assigns one
of four evidence statuses with a short rationale.

It is **not a new detector** — it sits on top of the existing pipeline.

---

## 2. The command to run

From the repo root, with no GPU and no model weights required:

```bash
python -m proof_of_concept.run_demo --all \
  --output proof_of_concept/outputs/report.md
```

Standard library only. Runs in ~3 seconds. Produces a Markdown report.

---

## 3. The file to open

Open `proof_of_concept/outputs/report.md` in Cursor's Markdown preview pane.
The report has four sections — one per scenario — each with:

- A per-run table (prediction, real/fake scores, confidence, runtime, error)
- The evidence status (`Stable` / `Contested` / `Uncertain` / `Failed/insufficient`)
- A one-sentence rationale generated by the rule layer
- The model's answer excerpt for each of the four runs

---

## 4. The scenario that best demonstrates the thesis contribution

Lead with **`contested`**. It is the only scenario that an isolated X2DFD
call cannot tell you about, and it shows the layer doing real work:

- Bare LLaVA (`none`) and the blending-only run both say **real**.
- The diffusion-only run and the combined run both say **fake**.
- The rationale flags this as `Contested` and tells you *why*:
  *"Sources of disagreement: adding experts flips bare-LLaVA verdict;
  blending and diffusion experts disagree."*

This is the thesis pitch in one paragraph: an isolated MLLM verdict can be
overconfident or wrong; comparing across expert settings exposes that, and
the layer pinpoints the source of the disagreement.

Backup talking points for the other scenarios are in section 5.

---

## 5. How to explain the four labels

| Status | When it triggers | What it means for the user |
|---|---|---|
| **Stable** | All four runs agree AND combined-experts confidence ≥ 0.70 | Verdict is robust to which experts are present; safe to act on. |
| **Contested** | At least two runs disagree on real/fake | Verdict depends on which experts are used. Rationale names which comparison flipped. Needs human review. |
| **Uncertain** | All runs agree but combined-experts confidence < 0.70 | The model is consistent across configurations but not confident. Borderline case; flag for review. |
| **Failed/insufficient** | Combined run errored, or ≥ 2 of 4 runs failed | The comparison cannot be made honestly — most often a CUDA/OOM or missing-weight error. The layer refuses to assign a verdict rather than guess. |

Decision order is **Failed → Contested → Stable → Uncertain** (first match
wins). `Contested` is checked *before* high-confidence `Stable` on purpose:
a confident-but-contested verdict is exactly the case the layer exists to
surface.

Thresholds (`CONF_HIGH = 0.70`, `CONF_LOW = 0.55` reserved) live at the top
of `proof_of_concept/evaluator.py` so they are auditable and tunable
without reading any logic. The whole rule set fits on one screen — chosen
deliberately over a learned aggregator so the disagreement logic can be
defended in the thesis discussion.

---

## 6. What remains to connect it to real X2DFD outputs

The mock fixtures shipped with the POC are byte-compatible with what
`eval/infer/runner.py` writes, so the parser does not change. To swap in
real outputs:

1. **Place the X2DFD weights** at the paths in `eval/configs/infer_config.yaml`:
   LLaVA-1.5-7B base, the LoRA adapter, the blending detector
   (`weights/blending_models/best_gf.pth`), and the diffusion detector
   (`weights/ours-sync/`). Links are in the top-level `README.md`.

2. **Run the launcher** — one command, no shell loop. It executes the four
   `--experts` settings, names each output `demo_<run>.json`, captures
   per-run wall-clock to `runtimes.json`, and (with `--render`) writes the
   Markdown report next to them:

   ```bash
   python -m proof_of_concept.launcher \
     --image /abs/path/to/image.png \
     --render
   ```

   The launcher prints a status line per run and tolerates failures: if
   e.g. the diffusion expert OOMs, the comparison is automatically marked
   `Failed/insufficient` in the report — exactly like the bundled `failed/`
   fixture.

3. **(Optional) Sanity-check first** with `--dry-run` to print the four
   commands the launcher *would* run, without touching the GPU or weights:

   ```bash
   python -m proof_of_concept.launcher --image /abs/path/to/image.png --dry-run
   ```

4. **Open the report:** the launcher writes `report.md` into the same
   `eval/outputs/poc/launcher_<timestamp>/` directory it created for the
   four result JSONs.

The parser already handles the runner's `Inference error: ...` failure
strings, so a real-world OOM during the diffusion run will show up as a
`Failed/insufficient` comparison — exactly the case the bundled `failed/`
fixture demonstrates today.

**Outstanding risks** worth naming up-front in the meeting:

- Weights download size and Baidu/Drive availability.
- GPU memory headroom for LLaVA + diffusion expert at the same time.
- `max_new_tokens=4` in the canonical X2DFD config produces only the literal
  word `fake` / `real`, which makes the *answer excerpt* column thin. A
  one-line YAML bump to `~64` gives sentence-length rationales for the demo
  without retraining anything.
